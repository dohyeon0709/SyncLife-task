# 설계 결정 (DECISIONS)

> 이 문서는 평가에서 **20%**를 차지합니다. "무엇을 했는지"보다 **"왜 그렇게 했는지"**를 봅니다.
> 아래 항목을 채워 주세요. 코드로 답이 되는 부분은 파일 경로를 함께 적어도 좋습니다.

## 1. 상태 구조

- **서버 상태**: TanStack Query 쿼리로 관리합니다. QueryClientProvider가 전역 캐시를 제공하고 useQuery로 읽습니다.
- **클라이언트 상태**: 별도의 클라이언트 상태 관리 라이브러리를 쓰지 않았습니다. 클라이언트에서 관리해야 할 상태의 양이 크지 않다고 판단해 useState만으로 충분하다고 생각했습니다.
- **버린 대안**: Zustand 등으로 클라이언트 상태까지 통합 관리하는 방법도 고려했지만 관리할 상태 종류가 적어 별도 라이브러리를 추가하는 비용이 이점보다 크다고 판단해 채택하지 않았습니다. 다만 이후 프로젝트 규모가 커지거나 여러 컴포넌트에 걸쳐 공유해야 할 클라이언트 상태가 많아진다면 그때 Zustand를 도입해도 괜찮다고 생각합니다.

## 2. 낙관적 업데이트 & 롤백

- 카드 이동 시 TanStack Query의 useMutation으로 처리합니다. onMutate에서 서버 요청을 보내기 전에 캐시 스냅샷을 저장하고 캐시를 낙관적으로 먼저 바꿉니다. 요청이 실패하면 onError에서 그 스냅샷으로 캐시를 되돌리고 화면 하단 토스트로 사용자에게 알립니다.
- 롤백은 서버 재조회가 아니라 이전 스냅샷 복원 방식을 택했습니다. 이 프로젝트는 5000개 태스크를 한 번에 GET하는 구조라 실패할 때마다 전체를 다시 불러오는 건 비용이 크고 실패 시점의 서버 상태는 요청 보내기 전과 동일하므로 스냅샷 복원만으로 롤백이 가능합니다.
- 성공 시에는 onSuccess에서 서버가 반환한 태스크 객체로 캐시의 해당 항목을 교체합니다. 서버 쪽에서 version이 1 증가해서 오기 때문에 이 교체를 통해 다음 요청부터는 항상 최신 version을 사용하게 됩니다.
- 생성·수정·삭제도 같은 useMutation + onMutate/onError/onSuccess 패턴을 재사용합니다. 생성은 서버가 id를 만들어주기 때문에 onMutate에서 임시 id(temp-...)로 낙관적 카드를 먼저 넣고 성공하면 onSuccess에서 서버가 준 진짜 태스크로 교체합니다.

## 3. 경쟁 상태(race condition)

- 늦게 도착한 오래된 응답이 최신 상태를 덮어쓰는 걸 막기 위해 카드별로 이동 순번(seq)을 매겨서 onSuccess, onError 응답이 왔을 때 그 카드의 최신 이동이 더 있다면 응답을 무시합니다.
- 버전 충돌(409)은 요청을 큐로 직렬화하는 대신 서버가 409 응답에 담아주는 payload.current.version으로 1회 자동 재시도하도록 했습니다. 큐 방식보다 코드가 단순하고 기존 useMutation 패턴을 재사용할 수 있어서 선택했습니다.
- **버린 대안**: AbortController로 이전 요청을 취소하는 방법은 사용하지 않았습니다. PATCH 결과가 항상 같지 않아 클라이언트가 취소해도 서버가 이미 처리했을 수 있어 상태가 어긋날 위험이 있습니다. 로컬에서 version을 미리 낙관적으로 증가시키는 방법도 고려했지만 이 프로젝트는 쓰기 실패율이 15%라 앞 요청이 실패하면 미리 올려둔 version이 오히려 불필요한 충돌을 늘릴 수 있어 사용하지 않았습니다.

## 4. 대량 데이터 성능 (5,000개)

- tanstack/react-virtual로 각 컬럼을 가상화했습니다. 컬럼에 태스크가 아무리 많아도 화면에 보이는 카드(+overscan 여유분)만 DOM에 렌더합니다. 개발자도구로 확인해보면 컬럼당 태스크가 1000개가 넘어도 실제 DOM에는 20개 안팎만 존재합니다.
- 검색과 우선순위 필터는 배열 filter 한 번으로 처리합니다. 5000개 정도는 이 정도 연산이 매 렌더마다 걸려도 체감 지연이 없어서 검색 디바운싱 같은 추가 최적화는 지금은 넣지 않았습니다.

## 5. 정답이 없던 결정들 (명세서에 명시되지 않은 항목)

- **네트워크 완전 단절** 시 동작을 어떻게 정의했나요? 왜?
- **409 충돌** 시 사용자에게 무엇을 보여주고 어떻게 해소하나요? 왜?
- 실패한 쓰기의 **재시도**: 자동 vs 수동 — 무엇을 택했고 이유는?

## 6. 그 외 트레이드오프 / 남은 한계

- 핵심 로직 테스트는 컴포넌트 통합 테스트 대신 순수 함수 추출 방식을 택했습니다. HTML5 네이티브 드래그 앤 드롭은 jsdom에서 흉내내기 어려워 이동 로직을 통합 테스트로 검증하기 힘들고 Board.tsx가 실제로 쓰는 배열 변경 로직(이동/생성/수정/삭제/필터/그룹핑)을 src/lib/tasks.ts로 추출해 그 함수들을 직접 테스트하는 쪽이 더 확실하고 빠르다고 판단했습니다.
