# 태스크 보드 견고화 (프론트엔드 채용 과제)

느리고 가끔 실패하는 mock 서버 위에서, 5,000개 규모의 태스크를 다루는 칸반 보드를
비동기 견고성(낙관적 업데이트·롤백·경쟁 상태 처리)을 중심으로 완성한 과제입니다.

- **배포 URL**: https://dohyeon0709.github.io/SyncLife-task/
- **설계 결정과 근거**: [DECISIONS.md](./DECISIONS.md)
- **AI 도구 활용 내역**: [AI_USAGE.md](./AI_USAGE.md)

## 실행 방법

```bash
npm install    # postinstall 에서 public/mockServiceWorker.js 자동 생성
npm run dev     # 개발 서버
npm test        # 유닛 테스트 (Vitest)
npm run build   # 타입체크 + 프로덕션 빌드
```

Node 18 이상 권장.

개발 중 실패를 강제로 재현하려면 `src/mocks/config.ts`의 `WRITE_FAILURE_RATE`를 1로 올리면
모든 쓰기가 실패합니다(롤백 검증용). 확인 후 기본값(0.15)으로 되돌려야 합니다.

## 기술 스택

- React 18, TypeScript(strict), Vite, Vitest
- **TanStack Query** — 서버 상태 관리, 낙관적 업데이트(`useMutation`)
- **TanStack Virtual** — 5,000개 이상 리스트 가상화
- MSW — mock API (제공된 스타터)
- 드래그 앤 드롭, 모달, 토스트는 별도 라이브러리 없이 직접 구현 (근거는 DECISIONS.md 참고)

## 구현 기능

### Priority 1 (필수) — 모두 구현

| 기능                   | 내용                                                                                |
| ---------------------- | ----------------------------------------------------------------------------------- |
| 로드 상태 처리         | 로딩 / 에러(재시도 버튼) / 빈 상태 3분기 (`Board.tsx`)                              |
| 낙관적 업데이트 & 롤백 | 이동·생성·수정·삭제 모두 낙관적 반영 → 실패 시 정확히 롤백 + 토스트 알림            |
| 경쟁 상태 처리         | 카드별 이동 순번(seq)으로 오래된 응답 무시, 409 시 최신 version으로 1회 자동 재시도 |
| 대량 데이터 성능       | react-virtual로 컬럼 가상화, 검색+우선순위/태그 다중 필터                           |
| 태스크 CRUD            | 생성(제목·우선순위 필수, 설명 선택)/수정/삭제(확인 다이얼로그)                      |
| 핵심 로직 유닛 테스트  | `src/lib/tasks.ts`(필터·그룹핑·이동·생성·수정·삭제 로직) 테스트 15개                |

### Priority 2 (권장) — 일부 구현

구현함:

- 409 충돌 처리 UX (이동: 자동 재시도 / 수정: 서버 최신 내용 갱신 후 알림)
- 실패 토스트에 재시도 버튼 (자동 백오프 재시도 대신 수동 재시도 + 편의 버튼)
- 네트워크 완전 단절 감지 배너
- 우선순위·태그 다중 필터
- 키보드 접근성: 카드 포커스/Enter로 수정 열기, 방향키로 카드 간 이동, 모달 포커스 트랩 + Esc 닫기

미구현 (사유):

- **검색 디바운싱**: 5,000개 배열 필터링이 1~2ms 수준이라 성능상 불필요하고, 인위적 지연이
  오히려 체감 지연을 만든다고 판단해 의도적으로 넣지 않음 (DECISIONS.md 참고)
- **키보드로 카드를 다른 상태로 이동**: 포커스 이동/모달 접근성까지는 구현했지만, 네이티브
  드래그 앤 드롭을 대체하는 별도 이동 UI까지는 시간상 구현하지 못함
- **다중 탭 동기화**: BroadcastChannel로 구현 가능하나 우선순위상 시간 배분에서 제외
